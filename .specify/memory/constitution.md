# Physical AI & Humanoid Robotics Textbook Constitution

## Core Principles

### I. Spec-Driven Content
All content creation, structure changes, or feature additions begin with a specification. No chapters or modules are added without a clear, approved spec defining the learning objectives and scope.

### II. Content Quality & Validation
"Testing" in this context means rigid validation: markdown linting, link checking, spell checking, and build verification. No broken links or malformed MDX are permitted in the `main` branch.

### III. Modular Structure
The textbook is composed of independent, loosely coupled modules. Directory structure is strict and semantic. Content is organized for reusability and clarity.

### IV. Documentation is the Product
The Docusaurus website *is* the deliverable. Clarity, accessibility, and user experience (UX) of the reader are paramount.

### V. Interactive Learning
Prefer interactive components (MDX, Mermaid diagrams, Tabs) over static text. Use the capabilities of Docusaurus 3.9 to enhance the learning experience.

## Project Metadata

- **Name**: Physical AI & Humanoid Robotics Textbook
- **Purpose**: To provide a comprehensive, interactive resource for learning physical AI and humanoid robotics.
- **Audience**: Students, Researchers, and Engineers in Robotics and AI.
- **Format**: Static Website (Docusaurus 3.9).

## Tech Stack Governance

### Framework
- **Core**: Docusaurus 3.9 (React-based static site generator).
- **Styling**: Standard Docusaurus styling (Infima/CSS).

### Directory Structure
Strict adherence to the following layout is mandatory:
```text
/
├── docs/
│   ├── modules/     # Core learning modules
│   ├── hardware/    # Hardware reference guides
│   ├── weeks/       # Weekly curriculum content
│   └── intro.md     # Landing page/Introduction
├── static/
│   └── img/         # Optimized images
├── src/             # Custom React pages/components
└── docusaurus.config.ts
```

### File & Naming Conventions
- **Files**: `kebab-case.md` or `kebab-case.mdx`.
- **Slugs**: Explicitly defined or matching filename (kebab-case).
- **Images**: Placed in `/static/img`, referenced as `/img/...`.
- **Sidebar**: Explicitly managed or strictly autogenerated via `_category_.json`.

### Metadata (Frontmatter)
Every documentation page MUST include:
```yaml
---
title: <Clear Title>
description: <Concise summary for SEO/Meta>
tags: [robotics, ros2, <topic>]
sidebar_position: <Integer for ordering>
---
```

## Content Writing Standards

### Formatting
- **Headings**: H1 (`#`) used ONLY for the page title. Use H2 (`##`) through H4 (`####`) for structure.
- **Code Snippets**: Must use fences with language tags.
  - ROS2/Python: `python`
  - C++: `cpp`
  - Config: `yaml`
  - Terminal: `bash`
- **Admonitions**: Use standard Docusaurus syntax `:::note`, `:::tip`, `:::warning`, `:::danger`.

### Diagrams & Media
- **Diagrams**: Prefer Mermaid.js code blocks (`mermaid`) for flowcharts and sequences over static images for editability.
- **Images**: Must be optimized (WebP/Compressed PNG). Alt text is mandatory for accessibility.

### Citations
- Use consistent linking to external resources.
- Academic citations should follow a standard footnote or reference section format.

## Plugin & Dependency Policy

### Approved Plugins
- `@docusaurus/theme-search-local` (or similar local search)
- `remark-math`, `rehype-katex` (for equations)
- `@docusaurus/plugin-content-docs` (standard)
- `@docusaurus/theme-mermaid`

### Policy
- **New Dependencies**: Must be approved via the spec/plan process (ADR required for heavy libs).
- **Licensing**: All assets (images, code) must be compatible with the project license (Open Source/Creative Commons).

## Agent Execution Rules (Gemini CLI)

1.  **Read First**: Gemini MUST read this Constitution before executing `sp.plan`, `sp.tasks`, or `sp.implement`.
2.  **Context**: Use `context7` tools for Docusaurus-specific API queries.
3.  **History**: Maintain accurate Prompt History Records (PHRs).
4.  **ADRs**: Suggest an Architecture Decision Record (ADR) for:
    - Adding new top-level directories.
    - Changing the sidebar strategy.
    - Integrating 3rd party UI libraries.

## Versioning & CI Rules

- **Git**: Feature branching model (`feature/module-name`). Main branch is always deployable.
- **Docusaurus Versioning**: (Currently single version). Follow semantic versioning for the project releases if enabled.
- **Deployment**: Automated to GitHub Pages.
- **CI Checks**:
  - `npm run build` (Must pass)
  - Markdown linting (recommended)
  - Broken link checker (`docusaurus build` catches most)

## Quality Gates

- **Structure**: Does the content fit the `/modules`, `/hardware`, or `/weeks` hierarchy?
- **Metadata**: Is frontmatter complete?
- **Assets**: Are images optimized and local?
- **Review**: Human review required for technical accuracy of robotics content.

## Expansion Guidelines

- **New Modules**: Create a new subfolder in `docs/modules/`. Add `_category_.json` for sidebar indexing.
- **RAG/Chatbot**: Future integration should use processed markdown content as knowledge base.
- **Hardware**: New robots go into `docs/hardware/<robot-name>/`.

## Governance

Amendments to this constitution (e.g., changing the stack, adding top-level folders) require a Pull Request and version bump.
